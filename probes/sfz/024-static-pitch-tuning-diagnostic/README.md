# 024 Static Pitch Tuning Diagnostic

This controlled probe checks static SFZ pitch behavior before implementing new
ConvertWithMoss/SFZv2 mappings in halionbridge.

The probe intentionally uses one sine single-cycle sample and one behavior
change per SFZ file. Use sforzando as the reference and the generated HALion
`.vstpreset` files as the converted output.

## Files

- `source/*.sfz`
  - Reference SFZ files for sforzando.
- `source/samples/sine_A3_single_cycle.wav`
  - Copied sample used by every SFZ.
- `halionbridge-build/*.vstpreset`
  - HALion presets generated by halionbridge.
- `plugnscript/halionbridge_sfz_static_pitch_tuning_generator.cxx`
  - Blue Cat Plug'n Script MIDI generator.

## SFZ Cases

| File | Opcode under test | Expected reference behavior |
| --- | --- | --- |
| `000_pitch_keycenter_57_reference.sfz` | `pitch_keycenter=57` | Baseline. |
| `001_pitch_keycenter_69_root_shift.sfz` | `pitch_keycenter=69` | Same played MIDI note should sound 12 semitones lower than baseline. |
| `002_tune_plus_100.sfz` | `tune=100` | One semitone above baseline. |
| `003_tune_minus_50.sfz` | `tune=-50` | 50 cents below baseline. |
| `004_transpose_plus_12.sfz` | `transpose=12` | One octave above baseline. |
| `005_transpose_minus_12.sfz` | `transpose=-12` | One octave below baseline. |
| `006_pitch_keytrack_0.sfz` | `pitch_keytrack=0` | Played notes should keep the same sample pitch. |
| `007_pitch_keytrack_50.sfz` | `pitch_keytrack=50` | Played notes should move at half normal pitch distance. |

## Manual Validation Loop

1. Load one `source/*.sfz` file in sforzando.
2. Load the matching generated `.vstpreset` in HALion 7.
3. Insert Blue Cat Plug'n Script before the sampler under test.
4. Load `plugnscript/halionbridge_sfz_static_pitch_tuning_generator.cxx`.
5. Start host transport.
6. Compare sforzando and HALion output with a tuner, spectrum view,
   oscilloscope, or by ear.

The generator repeats MIDI notes `57`, `69`, and `81` at velocity `100`.

## Current Expected halionbridge Result

The current converter is expected to preserve `pitch_keycenter` as HALion root
key. It is not yet expected to preserve `tune`, `transpose`, or
`pitch_keytrack`; those cases are diagnostic evidence for the next converter
mapping work.

Record each case as:

- matched,
- mismatched,
- ambiguous,
- or not testable.

## Observed Result

User validation result on 2026-06-13:

| Case | Result | Interpretation |
| --- | --- | --- |
| `000_pitch_keycenter_57_reference` | matched | Baseline root key behavior is correct. |
| `001_pitch_keycenter_69_root_shift` | matched | `pitch_keycenter` maps correctly to HALion root key. |
| `002_tune_plus_100` | mismatched; sforzando and HALion are clearly out of tune | HALion output is missing `tune=100`; implement cents tuning. |
| `003_tune_minus_50` | mismatched, but less strongly than `002` | HALion output is missing `tune=-50`; mismatch size follows the smaller source offset. |
| `004_transpose_plus_12` | mismatched; HALion is one octave too low | HALion output is missing positive transpose. |
| `005_transpose_minus_12` | mismatched; HALion is one octave too high | HALion output is missing negative transpose. |
| `006_pitch_keytrack_0` | mismatched; root note starts correctly, then HALion tracks upward while sforzando stays fixed | HALion output is missing disabled pitch key tracking. |
| `007_pitch_keytrack_50` | mismatched; HALion tracking sounds roughly double the sforzando pitch stepping | HALion output is missing half-rate pitch key tracking. |

Decision: preserve current `pitch_keycenter` mapping. Add converter/helper support
for `tune`, `transpose`, and `pitch_keytrack`, or emit structured diagnostics
until those mappings are implemented.
