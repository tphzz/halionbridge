# 049 Amplitude Envelope Post Release Regression

This probe follows `048-amplitude-release-two-segment-implementation-regression`.

Probe `048` confirmed that the converter-generated positive `ampeg_release`
mapping now matches the hand-authored curved two-segment candidates from
`047`. This probe regenerates the broader amplitude-envelope fixture from
`043` with the current converter, so the release fix can be checked against the
other static DAHDSR cases before moving on to filters.

## Files

- `source/*.sfz` are same-number sforzando references copied from probe `043`.
- `source/samples/amp_env_loop.wav` is the generated one-second seamless sine
  loop copied from probe `043`.
- `plugnscript/halionbridge_sfz_amp_envelope_generator.cxx` emits deterministic
  note input for both sforzando and HALion.
- `halionbridge-build/*.lua` are generated by `halionbridge convert sfz`.
- `halionbridge-build/*.vstpreset` are built through HALion and ready to load.

## Cases

- `000_default_release`: SFZv2 default `ampeg_release`.
- `001_release_1s`: full-level sustain with a one-second release; this should
  now use the verified curved two-segment release mapping.
- `002_attack_1s`: one-second attack, full sustain, medium release.
- `003_decay_sustain_40`: decay to `ampeg_sustain=40` with default
  `ampeg_decay_zero=1` behavior.
- `004_decay_zero_0`: same decay/sustain shape with `ampeg_decay_zero=0`.
- `005_delay_attack_hold`: delay, attack, hold, decay, sustain, and release in
  one case.
- `006_start_50`: nonzero `ampeg_start` into the attack/decay path.
- `007_release_zero`: explicit `ampeg_release=0`; this should remain an
  immediate final zero point.
- `008_sustain_zero`: decay to zero sustain, useful for checking whether the
  envelope should free-run or stay at the sustain point.

1. Put Blue Cat Plug'n Script immediately before sforzando or HALion.

   - `003_decay_sustain_40.sfz` vs `003_decay_sustain_40.vstpreset`,
   - `004_decay_zero_0.sfz` vs `004_decay_zero_0.vstpreset`,
   - `005_delay_attack_hold.sfz` vs `005_delay_attack_hold.vstpreset`,
   - `006_start_50.sfz` vs `006_start_50.vstpreset`,

Use the phase-inverted/null-test setup where practical. The key question is
whether `001` now matches like the verified release probes while `000`, `002`
through `008` remain at least as good as they were in `043`.

## Result

Generation and HALion build completed successfully on 2026-06-14: nine SFZ
files generated nine build scripts plus the helper module, and HALion saved
nine `.vstpreset` files.

Conversion emitted the same expected sfizz warning for `004_decay_zero_0.sfz`
as probe `043`: sfizz reports `ampeg_decay_zero` as unsupported, while
halionbridge's explicit-opcode overlay still reads it.

Generated Lua inspection confirmed:

- `001_release_1s` uses the verified curved two-segment release shape,
- `007_release_zero` keeps an immediate final zero point.

Manual validation on 2026-06-14:

- `001_release_1s`, `002_attack_1s`, and `007_release_zero` work well.
- `003_decay_sustain_40` and `004_decay_zero_0` have decay stages that are too
  slow in HALion.
- In `005_delay_attack_hold`, delay, attack, and hold match, but decay is too
  slow. Manually changing the HALion decay to about `200 ms` with UI
  `Curve=-10` gives a reasonable match.
- In `006_start_50`, the audible initial level behaves closer to `0` than to
  the generated HALion level `0.5`; the following decay has the same too-slow
  decay problem.
- In `008_sustain_zero`, HALion produces a release bump that is not present in
  sforzando. The generated positive-release early level is fixed at `0.35`,
  which is wrong after the envelope has already decayed to sustain level `0`.

Decision: keep the positive-release fix from probes `047` and `048`, but do
not close static amplitude envelopes yet. Use probe `050` to isolate decay
curve/timing and the sustain-zero release bump.
